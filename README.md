# Retail ETL Pipeline

Конвейер обработки данных розничных продаж: консолидация таблиц от нескольких поставщиков, унификация наименований, категоризация, анонимизация и аналитика — всё управляется через веб-интерфейс Streamlit или CLI.

---

## Архитектура

```
 Источники (xlsx / xls / csv / tsv)
          │
          ▼
 ┌──────────────────────────────────────────────────────────┐
 │                      pipeline.py                         │
 │                                                          │
 │  1. Чтение + маппинг колонок (fuzzy по ALIASES)         │
 │  2. Категоризация → target_category  (keyword-dict)      │
 │  3. Мэтчинг наименований → canonical_name  (fuzzy)       │
 │  4. Анонимизация (магазины / бренды / товары / цены)     │
 │  5. Объединение + опциональная агрегация                 │
 │  6. Сохранение merged_YYYYMMDDHHММ.xlsx                  │
 └──────────────────────────────────────────────────────────┘
          │
     ┌────┴────────────────┐
     │                     │
     ▼                     ▼
 Streamlit UI         Dagster + Kafka
 review_app.py        dagster_job.py
```

## Быстрый старт

### Веб-интерфейс (рекомендуется)

```bash
streamlit run review_app.py
```

Откроется браузер на `http://localhost:8501`. Все настройки — в боковой панели и на вкладке **Состояние**.

### CLI

```bash
python pipeline.py
```

Интерактивный выбор папок → обработка → сохранение результата.

### Dagster (оркестрация)

```bash
dagster dev -f dagster_job.py
```

UI на `http://localhost:3000`. Пайплайн также запускается командой `/merge` в Kafka-топике `etl-commands`.

## Веб-интерфейс (Streamlit)

### Обработка
Выбор файлов, переключатели **Категоризация** и **Обезличивание**, запуск пайплайна, скачивание результата в XLSX или CSV.

### Аналитика
BI-дашборд поверх результирующих таблиц:
- KPI-карточки: выручка, количество, себестоимость, маржа
- Динамика продаж (stacked area), тепловая карта по неделям/месяцам
- ABC/XYZ-анализ товаров
- Сравнение периодов
- Обнаружение аномалий (z-score)
- Прогноз продаж (линейная экстраполяция)
- Экспорт в Excel (многолистовой) и PDF

### Категории
Редактор списка целевых категорий (`target_categories.json`) и ручных правок категоризации (`category_overrides.json`).

### Файлы
Управление папками «Источник / Результаты / Архив»:
- Просмотр файлов с размерами
- Загрузка новых файлов
- Drag-and-drop перемещение между папками (физически на диске)
- Групповое скачивание ZIP
- Удаление группой

### Состояние
Визуальная форма конфигурации с подсказками, редактор JSON-файлов, очистка кэшей.

## Как работает категоризация

Двухпроходная классификация без LLM — работает мгновенно:

1. **Pass 1** — взвешенный Jaccard по словам исходных категорий поставщика и словам целевых категорий. Срабатывает, когда данные поставщика уже содержат читаемые категории.
2. **Pass 2** — сопоставление по кластерным keyword-словарям (`_PRODUCT_CLUSTERS`): если в названии товара есть «гель для стирки» → кластер `стирка` → целевая категория «стирка».

---

## Как работает мэтчинг наименований

Цель — унифицировать разные написания одного товара от разных поставщиков:

```
«Ariel Color 900мл»   \
«Ariel Colour 900мл»   → canonical: «Ariel Color 900мл»
«Ариель Колор 900 мл» /
```

Алгоритм (без LLM):

1. **Блокировка** по `(бренд, объём)` — сравниваем только внутри одного блока.
2. **Нормализация** — нижний регистр, транслитерация кирилл. → лат., очистка пунктуации.
3. **Схожесть** = `max(token_jaccard, seq_ratio × 0.9)` — Jaccard не зависит от порядка слов, SequenceMatcher ловит опечатки и латинские/кириллические варианты.
4. **Union-Find кластеризация** пар с similarity ≥ 0.78.
5. **Canonical** = самое частое название в кластере (при равенстве — самое длинное).

Результат пишется в колонку `canonical_name` и кэшируется в `name_matches.json`.

## Kafka

Запустить локально через Docker:

```bash
docker-compose up -d
```

| Топик | Назначение |
|---|---|
| `etl-commands` | команды запуска (`/merge`) |
| `etl-results` | результаты (путь к сохранённому файлу) |

```bash
python producer.py    # отправить /merge
python consumer.py    # читать результаты
```

## Конфигурация

`config.json` создаётся автоматически (в git не попадает):

```json
{
  "input_folder":          "data",
  "archive_folder":        "archive",
  "output_folder":         "merged",
  "category_folder":       "categories",
  "anonymize":             false,
  "categorize":            true,
  "name_match":            false,
  "aggregate_after_merge": false
}
```

Все поля редактируются в Streamlit UI (вкладка **Состояние**).

## Структура проекта

```
pipeline.py              ETL-ядро: чтение, трансформация, объединение, запись
category_matcher.py      двухпроходная keyword-категоризация
name_matcher.py          fuzzy-мэтчинг наименований
dagster_job.py           Dagster ops/job + Kafka-сенсор
review_app.py            Streamlit веб-интерфейс
review_cli.py            CLI-управляющий интерфейс
producer.py              Kafka-продюсер
consumer.py              Kafka-консьюмер
docker-compose.yml       Kafka (Zookeeper + Broker)
requirements.txt         зависимости
target_categories.json   список целевых категорий
category_overrides.json  ручные правки категоризации
name_matches.json        кэш мэтчинга + ручные правки

data/                    [git-ignored] входные таблицы
merged/                  [git-ignored] результаты
archive/                 [git-ignored] архив обработанных файлов
categories/              [git-ignored] источник категорий поставщика
```
