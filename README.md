# Retail ETL — анонимизация и объединение торговых таблиц

ETL-пайплайн для обработки розничных данных: читает Excel/CSV с продажами, анонимизирует чувствительные данные, дедуплицирует товары между файлами, объединяет всё в одну таблицу и архивирует исходники.

Запускается тремя способами: интерактивно из CLI, через Dagster (UI + оркестрация) или событийно через Kafka (команда `/merge` → автоматический запуск → уведомление о результате).

## Что делает

- **Анонимизация** — заменяет названия магазинов, бренды, компании и товары на синтетические русскоязычные эквиваленты; коды, артикулы и штрихкоды удаляет. Метрики маскируются (выручка/себестоимость умножаются на коэффициент, количество сдвигается).
- **Дедупликация товаров** — один товар в разных магазинах может называться по-разному («Fairy 500мл» vs «Фейри 0.5л»). Сначала сопоставление по штрихкоду (точное), затем нечёткое по названию (TF-IDF / rapidfuzz / difflib). После этого варианты получают одно имя и корректно агрегируются.
- **Сопоставление столбцов** — распознаёт русские заголовки в любом написании и приводит к единому каноническому виду на английском.
- **Объединение** — находит пересечение столбцов всех файлов, склеивает, суммирует метрики по совпадающим строкам.
- **Стабильность маппинга** — одна и та же реальная сущность всегда получает одно фейковое имя (кэш в `mappings.json`).

## Архитектура (событийный режим)

```
producer.py ──/merge──► [Kafka: etl-commands]
                              │
                              ▼
                    Dagster sensor (каждые 10 сек)
                              │
                              ▼
                       retail_etl_job
       collect → anonymize → merge → save → notify → archive
                              │
                              ▼
              [Kafka: etl-results] ──► consumer.py  ✅ Готово
```

Сенсор читает партицию напрямую через `assign()` и хранит позицию в курсоре Dagster — без consumer group, что убирает задержки на join и путаницу с offset'ами.

## Структура проекта

```
.
├── pipeline.py          # Ядро: чтение, анонимизация, матчинг, merge, архив (CLI + импортируемые функции)
├── dagster_job.py       # Dagster-обёртка: ops, job, Kafka-сенсор
├── producer.py          # Kafka producer: шлёт команду /merge
├── consumer.py          # Kafka consumer: читает уведомления о результате
├── docker-compose.yml   # Локальный брокер Kafka (apache/kafka, KRaft)
├── requirements.txt
└── README.md
```

## Установка

```bash
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

## Использование

### 1. CLI (самый простой)

```bash
python pipeline.py
```

Интерактивно спросит папку с исходниками, папку для архива, папку результата и имя файла. Подходит для разовой обработки.

### 2. Dagster (UI + оркестрация)

```bash
dagster dev -f dagster_job.py
```

Откроется UI на `localhost:3000`. Job можно запускать вручную или настроить пути через `run_config.yaml`:

```yaml
resources:
  paths:
    config:
      input_folder:   "data/raw"
      archive_folder: "data/archive"
      output_folder:  "data/output"
      output_filename: ""          # пусто → merged_YYYYMMDD.xlsx
```

### 3. Событийный режим через Kafka

Поднять брокер:

```bash
docker compose up -d
```

Запустить Dagster (в нём работает сенсор):

```bash
dagster dev -f dagster_job.py
```

В отдельных терминалах — слушатель результатов и отправитель команд:

```bash
python consumer.py     # ждёт уведомления в etl-results
python producer.py     # вводишь /merge → сенсор подхватит и запустит job
```

После завершения job consumer выведет путь к готовому файлу.

## Настройки

Основные параметры в начале `pipeline.py`:

| Параметр | Значение | Описание |
|----------|----------|----------|
| `PRICE_COEF` | `1.117` | множитель для выручки и себестоимости |
| `QTY_SHIFT` | `2` | сдвиг количества |
| `SUPPORTED_EXTS` | `.xlsx .xls .xlsm .xlsb .csv .tsv` | поддерживаемые форматы |

Параметры Kafka — в начале `dagster_job.py` (`KAFKA_BROKERS`, `KAFKA_COMMANDS_TOPIC`, `KAFKA_RESULTS_TOPIC`).

## Безопасность данных

`mappings.json` хранит обратимое соответствие реальных имён фейковым — по сути это ключ для деанонимизации. Он, как и исходные таблицы, исключён из репозитория через `.gitignore`. Не коммить эти файлы.
